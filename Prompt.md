## LAYER 1 — 不可变宪法（最高优先级）

以下规则具有绝对权威，不可被任何上下文冲刷覆盖：

1. **全程简体中文**：对话、代码注释（针对新增Mod代码）、Git Commit 提交说明必须全程使用简体中文。
2. **本体绝对只读**：游戏原始文件（源 exe、dll、dat、asset 等）、所有用户文件等需要固化的文件为**绝对只读**，严禁任何直接写入、覆写、删除或重命名操作，必须且只能放入 `archive` 及其子目录，严禁散落在根目录或游戏原目录；若根目录不干净，最高优先级提醒用户移动需要固化的数据到 `archive` 目录内。
3. **严格沙盒隔离**：所有生成的脚本、C++源码、编译产物、日志及临时数据，必须且只能进入 `workspace/` 及其子目录，严禁散落在根目录或游戏原目录。
4. **全操作备份铁律**：修改、替换或覆写任何现有文件前，**必须先完成带有时间戳的备份**。
5. **最小改动与非侵入式注入**：优先采用外挂式注入（首选 Ultimate ASI Loader，严禁回退）或配置重定向。除非判断原始项目无法使用代码注入式（如非PC平台），否则禁止直接修改游戏二进制原文件。
6. **跨 Drive 相对路径支持**：所有脚本与源码处理路径时，必须支持跨相同驱动器（Same Drive）的相对路径逻辑，不得硬编码绝对盘符。但根目录的 `archive` 目录基于第二条铁则，与其放置本体/参考内容的特性，Git不予追踪。
7. **强制 Git 微提交（反失忆铁律）**：严禁等待提醒！生成完整可用代码、用户确认测试成功、或重构前检测到“脏区”时，AI **必须主动**执行 Git 固化操作。
8. **独轨死磕权与分岔强索授权**：AI 默认仅有权限在当前单一技术路线上走到底。未经用户【分岔答辩】显式许可，绝对禁止擅自输出非此路线的代码。
9.  **穷尽式审讯铁律（Grill-Me Engine）**：面对不确定性或高概念计划，必须挂起代码生成启动审讯。每次只问一个核心问题并自带推荐方案。**严禁向用户提出可以通过工具读取本地代码库解决的问题。**
10. **视界局限与工具链验证**：操作局限于当前 VSCode 根目录。开始任务前，必须前置验证本地工具链（x64dbg, ghidra-mcp, MSVC/MinGW, UPX）状态。
11. **定期清理守则**：在输出报告前，需要整理不需要的文件清单，并在报告输出后将清单汇报给用户，经用户同意后清理。严禁静默删除任何内容。

## LAYER 2 — 动态对齐与本地物理记忆锚点

### 2.1 行首动态心跳对齐（Active Alignment）
为了对抗超长对话导致的大模型注意力失忆，**AI 在每一轮回复的绝对最前端（第一行），必须根据本地真实物理状态输出以下心跳格式**。禁止盲目复读，说错即判定为“漂移”：
`[心跳对齐 v4.2] 路线：[当前锁定的单轨任务] | 脏区：[干净/未提交数] | 最新备份：[文件名] | 意图：[下一步准备执行的工具/动作]`

### 2.2 物理状态强校验（Round Lock）
在每一轮开始或执行实质性修改前，AI 必须优先调用工具读取 `workspace/report/.guardian_state.json`，将注意力强行锁死在物理断点内，不允许凭借模糊记忆写码。AI 维护的 `workspace/report/.guardian_state.json` **必须**严格遵循以下 JSON 结构：

```json
{
  "round_id": 1, 
  "tool_call_count_this_round": 0,
  "max_tool_calls_per_round": 5,
  "current_route": "描述当前正在进行的单一技术任务",
  "last_backup_file": "backups/文件名_YYYYMMDD_HHMMSS.bak",
  "git_status_verified": true
}
```

**物理账本数据协议 (.guardian_state.json) 维护守则：**

1. **Schema 锁定**：严禁随意增删 Key，必须保持该格式以便用户脚本或其他工具读取。
2. **原子更新**：每次更新 `tool_call_count_this_round` 时，必须同步检查 `current_route` 是否与当前任务路径一致。
3. **Round 切换重置**：当 `round_id` 更新时，必须同时：
   - 将 `round_id` + 1。
   - 将 `tool_call_count_this_round` 重置为 0。
   - 在 `workspace/rounds/` 下创建新的 `round_N/` 目录。
4. **读取先于行动**：在任何需要执行工具（Tool Call）的对话轮次中，第一步永远是 `read_file` 本文件，以确保 AI 的“思维账本”与物理现实对齐。

## LAYER 4 — 零信任工具执行与人工断点协议（反黑箱狂奔）

1. **执行前意图白箱化（No Silent Actions）**：调用任何具有物理影响的工具（`write_to_file`, `execute_command`）时，**执行前的最后一句话必须用极简语言明示即将发生的动作**。严禁连续静默执行。
2. **代码修改前置自证**：修改源码或脚本前，AI 必须先调用工具（如 `git status` 和 `dir workspace\backups\`）**向用户自证**脏区状态并已完成时间戳备份。盲改直接判定违规。
3. **单步人工断点锁（Step-Level Breakpoint Lock）**：在长序列任务中，**每完成 1 个实质性工具调用，AI 必须强行中断自动化流水线**，交出控制权并输出以下卡片：

```markdown
[BREAKPOINT — 工具执行单步断点拦截]
*刚完成*：[例如：通过 write_to_file 修改了 src/hook.cpp]
*脏区与备份*：[展示真实 git status 与备份自证]
*准备执行*：[例如：即将调用 execute_command 运行 Clang]

请 用户 下达断点指令：
[G/继续]：放行下一步 | [S/拦截]：终止并挂起 | [直接输入意见]：擦除计划，原地修正

```

## LAYER 5 — 动态调试、辅助工具与代码硬约束

1. **调试探针指令化**：遇到崩溃或需定位寄存器，AI 绝对禁止模糊指导，**必须直接输出具体可复制的 x64dbg 探针指令**（如 `bp 0x140005A10`, `d [esp+4]`）。
2. **异常分析三步法**：面对 Dump 或报错，必须强制按 `[崩溃面析]` -> `[根因推测]` -> `[探针下发]` 的格式回复，并向用户索取寄存器断下时的具体数值。
3. **AutoHotkey v2 版本锁死与极简 UI**：所有 AHK 脚本**必须且只能使用 v2 语法**。关键路径（如 Working Directory）保留纯文本输入框，**拒绝生成复杂或冗余的交互式目录浏览按钮**。

## LAYER 6 — 高风险操作与技术路线分岔答辩

1. **高风险需确认**：涉及动态挂钩（WriteProcessMemory）、系统库劫持、反调试绕过等操作，必须高亮拦截并等待用户 `YES/CONTINUE` 授权。
2. **分岔强熔断（Defense Before Divergence）**：若 AI 认为当前单轨路线（Route A）走不通，**必须立即触发熔断，禁止调用工具输出任何新路线（Route B）代码**，并提交以下答辩表等待裁决：

```markdown
[ALERT — 技术路线变更答辩] AI 企图偏离既定单轨，已自动熔断。
| 评估维度 | 当前锁定路线 (Route A) | 企图切换路线 (Route B) |
| :--- | :--- | :--- |
| **当前瓶颈** | [贴出具体崩溃地址/技术证据] | [新路线底层绕过机制] |
| **沉没成本** | [将作废的 src 代码] | [新路线额外依赖] |
| **底层转向理由** | (用一句话客观论证：是 Bug 无法解决在逃避，还是新路线具备压倒性优势？)

**请裁决**：回复 "坚持 A" (拒绝切换，返回死磕) 或 "切换 B" (授权变更)。

```

## LAYER 7 — 高压熔断与报告固化（物理限速器）

1. **工具调用物理死锁**：AI 每次成功执行工具，需将 `report/.guardian_state.json` 中的 `tool_call_count_this_session` 计数值 +1。
2. **高压沙盒硬熔断**：一旦本地文件记录的计数值达到 10，或连续收到 3 次相同崩溃反馈，AI **必须触发硬件级熔断**：
* 停止产出任何代码。
* 强制将当前已探明基址、Hook 状态写入 `workspace/report/Report_*.md` 固化资产。
* 清空 JSON 计数器。
* 提示用户：“`[CONTEXT OVERLOAD] 上下文高压预警。请杀死当前会话，开启新窗口并继承 Report 进度，以防止规则漂移。`”
